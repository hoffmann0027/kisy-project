## 0. Честная рамка угроз (прочитать до первой строки кода)

**Кого защищаем и от кого.** Модель угроз — «honest-but-curious / скомпрометированный
хостер»: сервер (Render, его БД, дампы, бэкапы, сотрудники, тот, кто получит доступ
к диску/дампу) не должен читать тексты, файлы и метаданные-содержимое.

**Что E2EE даёт:** сервер хранит только шифротекст сообщений и вложений; ключи
существуют только на устройствах пользователей. Дамп БД = бесполезный набор байт.

**Что E2EE НЕ прячет (метаданные):** кто с кем в чате, время и размер сообщений,
кто онлайн. Это уменьшаемо (см. §7), но полностью не устраняется без анонимной сети.

**⚠️ Фундаментальное ограничение веб-E2EE — и как мы его снимаем (РЕШЕНО).**
Тот же сервер, что хранит шифротекст, **отдаёт браузеру JavaScript, который делает
шифрование**. Скомпрометированный сервер может один раз отдать вредоносный JS, который
украдёт ключ или открытый текст на клиенте. Транспортное TLS от этого не спасает.
Поэтому «сервер физически не сможет» на чистом вебе недостижимо на 100%.

**Решение заказчика: клиент — desktop-приложение на Tauri** с подписанным,
воспроизводимым билдом. Код шифрования **не приходит с сервера при каждом заходе** —
он внутри установленного, подписанного бандла. Это снимает главное ограничение: сервер
отдаёт только данные (шифротекст) и directory-API ключей, но не исполняемый код клиента.
Требования к реализации:
- **Tauri-обёртка** над существующим React/TS-фронтом; крипто-ядро (§1) — общий код,
  **не зависящий от рантайма** (работает и в вебе для разработки, и в Tauri в проде).
- **Подпись билда** (code signing) для Windows/macOS + **воспроизводимый билд**
  (reproducible build), чтобы пользователь/аудитор мог сверить, что установленный
  бинарь собран из опубликованного исходника.
- Обновления — только через подписанный updater Tauri (проверка подписи апдейта).
- Веб-версию (если останется) держать лишь как dev/резервный доступ и честно помечать
  в UI как «менее защищённый режим» (к ней применяется весь §6).
- Никакой загрузки удалённого JS/eval в Tauri-клиенте; CSP Tauri-окна — строгий
  (`script-src 'self'`), внешние ресурсы запрещены.

Криптоядро строим независимым от рантайма с самого начала — §6 (Trusted Types, SRI,
non-extractable ключи) остаётся актуальным и для Tauri-webview, и для веб-резерва.

**Принцип для всего плана:** приватные ключи — `non-extractable` где возможно; сервер
никогда не логирует шифротекст-в-открытом, ключи или пароли; поиск и парсинг упоминаний
переезжают на клиент; всё, что раньше требовало «сервер читает текст» — пересматривается.

## 1. Крипто-архитектура (спроектировать до кода)

### 1.1 Примитивы
Использовать **libsodium** (`libsodium-wrappers-sumo`) на клиенте — проверенная,
без «сделай сам». Набор:
- **X25519** — согласование ключей (ECDH);
- **Ed25519** — подписи (идентичность, верификация);
- **XChaCha20-Poly1305** — симметричное AEAD (тело сообщения, файлы);
- **Argon2id** (из libsodium) — вывод ключа из пароля на клиенте;
- **BLAKE2b** — хеши/фингерпринты.

Никакой собственной криптографии, никакого AES-ECB, никаких статических nonce.
Nonce — всегда случайный (XChaCha допускает 24 байта → безопасно рандомно).

### 1.2 Ключи пользователя
На регистрации/первом входе клиент генерирует:
- **Identity key** — Ed25519 (подпись) + X25519 (ECDH). Долгоживущий.
- Набор **prekeys** (одноразовые X25519) — для X3DH-рукопожатия оффлайн-получателю.

**Хранение приватного ключа:**
- локально — в IndexedDB как `non-extractable` CryptoKey, где формат позволяет;
- **РЕШЕНО (мультидевайс): QR-линковка устройств + опциональный зашифрованный бэкап.**
  - Основной путь — **линковка нового устройства по QR** (как WhatsApp): приватные
    ключи переносятся напрямую между своими устройствами через одноразовый
    зашифрованный канал (QR несёт эфемерный ключ; сервер выступает только «немым»
    ретранслятором шифротекста). Пароль-бэкап для этого не нужен.
  - **Опциональный** зашифрованный «key bundle» (приватные ключи + при желании история)
    под ключом из пароля/фразы восстановления (Argon2id **на клиенте**) кладётся на
    сервер как BYTEA-блоб — как страховка от потери всех устройств. Сервер видит только
    шифротекст бэкапа; стойкость завязана на пароль → требовать сильную passphrase и
    честно предупреждать пользователя. По умолчанию — выключено, включается осознанно.
  - Пароль аутентификации на сервер в открытом виде не уходит (он и так Argon2id-хеш);
    ключ бэкапа выводится из **отдельной** passphrase восстановления, не переиспользуя
    пароль входа.

### 1.3 Личные чаты (1:1)
**Рекомендуемо: X3DH + Double Ratchet** (протокол Signal): forward secrecy
(компрометация ключа не раскрывает прошлые сообщения) + post-compromise security.
Взять готовую реализацию (`libsignal` / проверенный порт), не писать ратчет вручную.

**Упрощённая альтернатива (если сроки жмут):** один симметричный ключ на чат
(`crypto_secretbox` / sealed box на identity-ключах). Проще, но без forward secrecy.
Явно зафиксировать компромисс в `docs/security.md`. Рекомендация — не идти этим путём
для «корпоративной» подачи; FS ожидаем.

### 1.4 Групповые чаты (важно: тут ролевая модель 1–10)
Группы — самое сложное. Варианты:
- **MLS (RFC 9420)** — современный стандарт групповой E2EE, эффективные add/remove
  участников, FS. Идеален под вашу модель «добавили/убрали по уровню допуска». Взять
  готовую библиотеку MLS.
- **Sender Keys** (схема групп Signal) — проще MLS, каждый шлёт под своим sender-key,
  розданным попарно. При исключении участника — ротация ключа.

Рекомендация: **MLS**, если найдётся зрелая JS/WASM-реализация; иначе Sender Keys.
Ключевое: при **исключении/понижении** пользователя из группы — обязательная **ротация
группового ключа** (иначе он расшифрует будущие сообщения). Это связывает крипту с
существующей `internal/access` / RBAC-логикой — заложить хук.

### 1.5 Защита от подмены ключей сервером (MITM)
Сервер — недоверенный «справочник» открытых ключей. Он может подсунуть свой открытый
ключ. Обязательно:
- **TOFU** (trust-on-first-use) + **предупреждение при смене ключа собеседника**
  («ключ безопасности изменился»);
- **Safety numbers / фингерпринты** (как в Signal) — сравнение вне канала (QR при
  встрече, голос). UI для верификации контакта.
- Публикуемые prekeys подписываются identity-ключом (Ed25519) — клиент проверяет подпись.

## 2. Изменения на бэкенде (Go / PostgreSQL)

Сервер перестаёт понимать содержимое. Конкретно:

### 2.1 Схема БД (новые миграции — писать вручную, **не** `migrate dev`; см. память `avoid-prisma-migrate-dev` по духу: применять `migrate deploy` на выверенной миграции)
- `messages`: заменить `text TEXT` на:
  - `ciphertext BYTEA NOT NULL` (тело),
  - `nonce BYTEA NOT NULL`,
  - `sender_key_id` / `ratchet_header` (для группового/ратчет-контекста),
  - `alg SMALLINT` (версия схемы шифрования — для миграций крипты),
  - `content_type SMALLINT` (текст/файл/системное) — но **не** сам текст.
  - `reply_to` (UUID) — можно оставить в открытом виде (это просто ссылка).
- `attachments`: `data BYTEA` теперь **шифротекст файла** (шифруется на клиенте до
  загрузки); имя файла, mime, размер-«как есть» → тоже шифровать (класть в
  зашифрованный метаблоб). Оставить только несекретные технические поля.
- Написать `up`/`down`; для существующих данных — либо разовая клиентская
  ре-шифровка при миграции, либо (проще для прототипа) стартовать E2EE с чистой БД.
  **Спросить**, есть ли прод-данные, которые нельзя терять.

### 2.2 Поиск (ломается E2EE — решить осознанно)
`search_index` + `to_tsvector('russian', content)` требует открытый текст → **убрать
индексацию содержимого сообщений на сервере**. Замена:
- **клиентский поиск** по локально расшифрованному кэшу сообщений (IndexedDB) — рекомендация;
- (продвинуто, позже) blind-index/searchable encryption — сложно, отдельный этап.
- Поиск по пользователям/группам (имена) может остаться серверным, если имена не секретны.

### 2.3 Push-уведомления (VAPID уже есть)
Сервер не может вложить текст в пуш. Слать **content-less** пуш («новое сообщение»),
клиент по нему подтягивает и расшифровывает. Убрать любой открытый текст из payload пуша.

### 2.4 Что остаётся серверным (это НЕ конфиденциальность контента, не трогать без нужды)
Контроль доступа (RBAC/clearance `internal/access`), аудит-лог, rate-limit, маскировка
404 — остаются. Сервер по-прежнему решает «кому можно в чат», просто не читает *что* там.

### 2.5 Логи и ошибки
Пройтись `grep` по бэкенду: убедиться, что нигде не логируется тело сообщения/вложения.
Ключей на сервере нет по определению — проверить, что клиентские блобы ключей не пишутся
в логи/аудит в открытом виде.

## 3. Изменения на фронтенде (React/TS)

- Новый слой `shared/crypto/` (или `entities/crypto`): обёртка над libsodium, генерация
  и хранение ключей (IndexedDB, `non-extractable`), X3DH/ратчет или MLS-клиент.
- Шифрование **перед** отправкой в `shared/api` и `shared/ws`; расшифровка **после**
  получения, до попадания в стор. Сеть/стор/сервер видят только шифротекст.
- Вложения: шифровать файл в браузере (стримингово для больших) до upload; расшифровка
  на скачивании. Превью генерировать локально.
- UI верификации контакта (safety number/QR), баннер «ключ изменился».
- Клиентский полнотекстовый поиск по расшифрованному кэшу.
- Ничего секретного в `localStorage` (доступен любому JS) — только IndexedDB + по
  возможности non-extractable ключи.

- Линковка нового устройства по QR (перенос/выдача ключей между своими устройствами).

  (это плата за то, что сервер не может помочь — он ничего не видит).

## 5. Верификация E2EE (обязательный тест-этап)
Автотесты-доказательства, что сервер не видит контент:
- e2e-тест: отправить сообщение → **напрямую в БД** прочитать строку → убедиться, что
  `ciphertext` не содержит открытого текста и не расшифровывается без клиентского ключа.
- тест ротации группового ключа при исключении участника (старый участник не читает
  новые сообщения).
- тест «сервер подменил открытый ключ» → клиент показывает предупреждение.
- тест: пуш не содержит открытого текста; лог-дамп не содержит контента.

## 6. Максимизация доверия к веб-доставке (снижаем риск из §0)
Критично: **XSS = смерть E2EE** (инъекция JS крадёт ключи). Поэтому:
- CSP ужесточить до предела: `script-src 'self'` без inline (уже есть — проверить, что
  нет `unsafe-inline`/`unsafe-eval`), `object-src 'none'`, `base-uri 'self'`.
- **Trusted Types** (`require-trusted-types-for 'script'`) — блокирует DOM-XSS.
- **Subresource Integrity (SRI)** на все скрипты/чанки; проверить хеш libsodium-бандла.
- Запретить `dangerouslySetInnerHTML`; санитайз любого пользовательского HTML/markdown.
- **Supply chain:** зафиксировать версии (lockfile), `npm audit` в CI (уже есть),
  добавить SBOM, проверять целостность зависимостей, минимизировать число зависимостей
  в крипто-пути.
- Ключи — `non-extractable` CryptoKey, чтобы даже инъектированный JS не мог их выгрузить
  (шифровать/подписывать можно, экспортировать — нет).
- (Опция, сильный ход) вынести клиент в **Tauri/Electron** или расширение с
  воспроизводимым билдом — тогда §0-ограничение снимается. **Спросить заказчика.**

## 7. Минимизация метаданных (опционально, после основного)
- «Sealed sender» — скрыть отправителя от сервера в 1:1.
- Padding сообщений до фикс-размеров (прячет длину).
- Не хранить точные `read receipts`/online-timestamps дольше нужного.
- Явно задокументировать в `docs/security.md`, какие метаданные сервер всё же видит.

## 8. Общий hardening от взлома и кражи (дополнить существующий `docs/security.md`)
Многое уже сделано (Argon2id, JWT+ротация, RBAC, аудит, TLS 1.3, rate-limit, CSP,
маскировка 404). Добавить/проверить:
- **Аутентификация:** WebAuthn/passkeys и/или TOTP-2FA; экран «мои устройства/сессии»

- **At-rest шифрование БД** на стороне провайдера (defense-in-depth, хотя контент уже

- **Секреты:** только env/.env, ротация JWT-ключа, TURN-секрет уже эфемерный — оставить.
- **Заголовки/транспорт:** HSTS preload, TLS 1.3 — есть; проверить `Permissions-Policy`.
- **DoS/брутфорс:** есть (Nginx limit_req + Redis + блокировка) — прогнать нагрузочный тест.
- **Аудит/логи:** убедиться, что стек-трейсы и контент не утекают клиенту (есть) и в логи.
- **Зависимости/CI:** govulncheck + npm audit (есть) → добавить проверку на секреты в
  коммитах (gitleaks), Dependabot.
- **Финальный шаг:** обновить STRIDE-модель в `docs/security.md` под E2EE (новые
  доверительные границы: сервер теперь *вне* границы доверия для контента) и заказать/
  прогнать внешний аудит или `/security-review`.

## 9. Порядок этапов для Claude Code (каждый — собираемый и проверяемый)
1. **Проектирование крипты** — выбрать 1:1-протокол (Double Ratchet) и групповой (MLS
   vs Sender Keys), зафиксировать в `docs/e2ee-design.md`. Ответить на вопросы «спросить».
2. **Крипто-ядро на клиенте** — libsodium-обёртка, генерация/хранение ключей, юнит-тесты
   векторами.
3. **Схема БД + бэкенд** — миграции (ciphertext/nonce/alg), приём/выдача шифротекста,
   убрать серверную индексацию контента.
4. **1:1 E2EE** — X3DH + Double Ratchet, отправка/приём, верификация контакта.
5. **Группы E2EE** — групповой протокол + ротация ключа на add/remove (связать с RBAC).
6. **Вложения E2EE** — клиентское шифрование файлов, метаблоб.
7. **Мультидевайс/бэкап** — линковка QR, зашифрованный бэкап (если решили делать).
8. **Клиентский поиск + content-less push** — замена серверных фич.
9. **Веб-доставка hardening** — Trusted Types, SRI, CSP-аудит, non-extractable ключи.
10. **Тесты-доказательства E2EE** (§5) + обновить `docs/security.md` + `/security-review`.

## 10. Решения заказчика и оставшиеся вопросы

**Принято (2026-07-10):**
1. **Клиент — desktop-приложение на Tauri** (подписанный, воспроизводимый билд). Снимает
   фундаментальное ограничение §0. Веб — только dev/резерв, помеченный как менее защищённый.
2. **Мультидевайс — QR-линковка + опциональный зашифрованный бэкап** под отдельной
   passphrase (по умолчанию выключен). См. §1.2.
3. **Групповой протокол (MLS vs Sender Keys) — решить на этапе 1** после оценки зрелости
   доступных JS/WASM-библиотек MLS. Рекомендация по умолчанию: MLS, если есть зрелая
   реализация; иначе Sender Keys с ротацией ключа на add/remove.

**Ещё уточнить до этапа 3:**
4. **Есть ли прод-данные**, которые нельзя терять при переходе на E2EE? Если да — нужен
   этап клиентской ре-шифровки; если нет — стартуем E2EE с чистой БД (проще).
5. **Серверный поиск по сообщениям** — подтвердить отказ (замена клиентским поиском по
   расшифрованному кэшу). Для E2EE это обязательный компромисс.
