# KISY — Дизайн сквозного шифрования (E2EE)

> Этап 1 плана `docs/plan-e2ee-and-hardening.md`. Здесь зафиксированы все
> протокольные решения. Последующие этапы (2–10) реализуют этот документ.
> Дата: 2026-07-10.

---

## 1. Главное решение: один протокол — MLS (RFC 9420) для 1:1 и групп

**Решение.** И личные чаты (1:1), и группы работают поверх **MLS (Messaging
Layer Security, RFC 9420)**. Личный чат — это MLS-группа из двух участников
(из всех устройств двух пользователей, см. §4).

**Библиотека: [`ts-mls`](https://github.com/LukaJCB/ts-mls)** (MIT, чистый
TypeScript, v1.6.x, активно поддерживается — 36 релизов, последний март 2026;
проходит официальные RFC-тест-векторы и interop-тесты).

**Почему не X3DH + Double Ratchet для 1:1 (обоснование отступления от §1.3 плана):**

1. **Нет зрелой браузерной реализации Signal-протокола.** Официальный
   `@signalapp/libsignal` даёт TypeScript-биндинги только через нативный
   Node-модуль (Neon) — в браузере/WebView не работает.
   `libsignal-protocol-javascript` архивирован самим Signal.
   Порты (`libsignal-protocol-typescript`, свежие WASM-обёртки v0.1.x) — либо
   заброшены, либо младенческого возраста. Писать ратчет вручную план прямо
   запрещает — и это правильно.
2. **Один протокол вместо двух** — меньше поверхности атаки, один формат
   ключей, один directory-API, одна логика верификации. Ратчет + MLS удваивают
   крипто-код, который надо тестировать и поддерживать.
3. **MLS для 2 участников даёт то же, что и Double Ratchet:** forward secrecy
   (секреты эпох удаляются после использования) и post-compromise security
   (регулярные Update-коммиты обновляют leaf-ключи). Так делают Wire, Cisco
   Webex и Matrix (переход на MLS) — 1:1 как малая группа.
4. **Ролевая модель KISY (1–10) ложится на MLS нативно:** исключение/понижение
   участника = Remove-коммит → новая эпоха → автоматическая ротация группового
   секрета. Ровно то, что требует §1.4 плана.

**Риск и его фиксация:** `ts-mls` не проходил формальный аудит (как и любая
другая доступная JS/WASM-реализация MLS, включая обёртки OpenMLS, которых в
npm попросту нет). Компенсация: версия пинится в lockfile; наши обёртки
изолируют ts-mls за интерфейсом `shared/crypto/` (замена на OpenMLS-WASM в
будущем не трогает остальной код); тест-векторы RFC гоняются в нашем CI;
факт отсутствия аудита фиксируется в `docs/security.md`.

**Ciphersuite:** `MLS_128_DHKEMX25519_CHACHA20POLY1305_SHA256_Ed25519`
(X25519 + ChaCha20-Poly1305 + Ed25519) — в точности примитивы из §1.1 плана.
PQ-гибрид (X-Wing) ts-mls тоже умеет — оставляем как будущий апгрейд через
поле `alg`.

## 2. Разделение труда: MLS vs libsodium

MLS закрывает *сообщения*. Остальную клиентскую крипту делает
**libsodium** (`libsodium-wrappers-sumo`):

| Задача | Примитив libsodium |
|---|---|
| Шифрование вложений (стриминговое) | `crypto_secretstream_xchacha20poly1305` |
| Ключ бэкапа из passphrase | `crypto_pwhash` (Argon2id) |
| Fingerprint / safety numbers | `crypto_generichash` (BLAKE2b) |
| QR-линковка устройств (эфемерный канал) | `crypto_kx` (X25519) + `crypto_secretbox` |
| Случайность | `randombytes_buf` |

Никакой собственной криптографии; nonce — всегда случайные 24-байтовые
(XChaCha20) либо внутри MLS-фрейминга (там nonce детерминируется ключевым
расписанием эпохи — это часть RFC, не наша забота).

## 3. Ключи и идентичность

### 3.1 Устройство — единица доверия
Ключи привязаны к **устройству**, не к пользователю:

- **Device identity key** — Ed25519, генерируется при первом входе на
  устройстве, приватная часть **никогда не покидает устройство**.
- **MLS key packages** — одноразовые «предключи» (аналог prekeys из X3DH):
  устройство генерирует пачку (например, 50 штук), подписывает своим
  identity-ключом и выгружает на сервер. Сервер выдаёт по одному желающему
  добавить это устройство в группу и удаляет использованные.
- **Пользователь = множество устройств.** User identity — это набор
  device-ключей, связанных взаимными подписями (cross-signing, §4).

### 3.2 Хранение приватных ключей
- IndexedDB; всё, что формат позволяет — как `non-extractable CryptoKey`
  (WebCrypto Ed25519/X25519 поддерживают non-extractable). Материал, который
  обязан быть байтами (state MLS-групп), шифруется «локальным ключом
  устройства» — non-extractable AES-GCM CryptoKey в IndexedDB: инъектированный
  JS может *попросить* расшифровать, но не может *выгрузить* ключ.
- Ничего секретного в `localStorage`.
- В Tauri-клиенте (позже) локальный ключ дополнительно уезжает в
  OS keychain (плагин `tauri-plugin-stronghold`/keyring) — план §0.

### 3.3 Доверие ключам: TOFU + safety numbers
Сервер — недоверенный справочник. Обязательно (план §1.5):
- **TOFU**: первый увиденный набор устройств собеседника запоминается локально.
- **Баннер «ключ безопасности изменился»** при любом изменении набора
  устройств/ключей собеседника; сообщения не блокируются, но чат помечается.
- **Safety number** пары пользователей: `BLAKE2b(sorted(fingerprints всех
  identity-ключей обоих))` → 60 цифр + QR. Экран сравнения вне канала.
- Key packages подписаны device identity-ключом; identity-ключ нового
  устройства — подписан старым устройством (§4). Клиент проверяет всю цепочку.

## 4. Мультидевайс: QR-линковка без экспорта identity-ключей

Уточнение §1.2 плана: приватный identity-ключ **не переносится** — это и
безопаснее, и стандартно (так работают companion-устройства WhatsApp/Signal):

1. Новое устройство генерирует **свои** ключи и показывает QR: эфемерный
   публичный X25519-ключ + одноразовый код.
2. Старое устройство сканирует QR, устанавливает шифроканал (`crypto_kx`,
   сервер — немой ретранслятор шифротекста), проверяет короткий код на обоих
   экранах.
3. Старое устройство **подписывает** identity-ключ нового (cross-signing
   vouch) и публикует подпись в directory. Для остальных участников новое
   устройство теперь «то же лицо» — баннер смены ключа не срабатывает.
4. По тому же каналу опционально передаётся история сообщений (расшифрованный
   локальный кэш, пере-зашифрованный каналом линковки).
5. Каждое устройство добавляется отдельным листом во все MLS-группы
   пользователя (Add-коммит), при отвязке — Remove-коммит (ротация эпох).

**Бэкап (опция, по умолчанию выключен):** ключи устройства + история →
`crypto_secretstream` под ключом из **отдельной** recovery-passphrase
(Argon2id `MODERATE`+ на клиенте) → BYTEA-блоб на сервер. UI честно
предупреждает: забыл passphrase — история невосстановима.

## 5. Роль сервера: Delivery Service + Auth Service

Сервер перестаёт понимать контент и играет две MLS-роли:

- **Delivery Service (DS):** хранит и раздаёт key packages; фан-аут
  Welcome/Commit/Proposal и application-сообщений (наш существующий WS);
  хранение истории как шифротекст.
- **Authentication Service (AS):** привязка `user_id ↔ device ↔ ключи` через
  существующую JWT-аутентификацию при загрузке ключей. Но клиенты серверу не
  верят на слово — TOFU + подписи цепочки устройств (§3.3).

**Что остаётся серверным (план §2.4):** RBAC/clearance (`internal/access`),
членство в группах, аудит-лог, rate-limit, маскировка 404. Сервер решает
«кому можно в чат», крипта гарантирует «что written ≠ readable» для лишних.

### 5.1 Согласование членства с RBAC (ключевой хук)
Источник истины о членстве — сервер (RBAC). Крипто-членство (MLS-лист) обязано
следовать за ним:

- На `AddMember`/`RemoveMember`/понижение уровня сервер пишет запись в
  `e2ee_membership_queue` и шлёт WS-событие `e2ee.membership_changed`.
- Первый онлайн-клиент группы **с правом коммита** (владелец/админ, при их
  отсутствии — любой участник; MLS позволяет любому члену коммитить)
  забирает запись, делает Add/Remove-коммит, подтверждает обработку.
- Сервер валидирует порядок эпох (монотонный `epoch` на группу) и перестаёт
  доставлять сообщения исключённому сразу (RBAC), а крипта гарантирует, что
  даже украденный из БД шифротекст новых эпох он не расшифрует.
- Тест §5 плана: исключённый участник не читает сообщения новой эпохи.

### 5.2 Forward secrecy и «догрузка истории»
MLS даёт FS: старые эпохи не восстановимы из текущего состояния. Следствие —
**сервер хранит шифротекст, который новый участник/устройство расшифровать не
может**. История для нового устройства передаётся только линковкой (§4) или
бэкапом. Для нового участника группы история недоступна вовсе (фича, не баг —
зафиксировать в UX). Существующее устройство хранит расшифрованный кэш в
IndexedDB (шифруется локальным ключом, §3.2) — по нему работает поиск.

## 6. Схема данных (контракт для этапа 3)

### 6.1 Изменения `messages`
```sql
ALTER TABLE messages
  ADD COLUMN ciphertext BYTEA,          -- MLS PrivateMessage (фрейминг RFC 9420)
  ADD COLUMN alg        SMALLINT,       -- 1 = MLS ciphersuite §1; для миграций крипты
  ADD COLUMN epoch      BIGINT,         -- MLS-эпоха (валидация порядка, отладка)
  ADD COLUMN content_kind SMALLINT;     -- 1 текст, 2 вложение, 3 системное — БЕЗ содержимого
-- text TEXT → удаляется (см. §8: прод-данных нет, стартуем с чистой E2EE-схемы)
```
Отдельный столбец `nonce` не нужен: nonce внутри MLS-фрейминга. `reply_to`,
`chat_id`, `sender_id`, `created_at` остаются открытыми (метаданные, §7).
Реакции остаются открытыми (emoji на message_id — осознанный компромисс
метаданных, зафиксировать в security.md). Упоминания (`mentions`) переезжают
внутрь шифротекста (парсит клиент).

### 6.2 Новые таблицы
```sql
e2ee_devices        (id, user_id, name, ed25519_pub BYTEA, signed_by UUID NULL,
                     signature BYTEA NULL, created_at, revoked_at)
e2ee_key_packages   (id, device_id, key_package BYTEA, consumed_at NULL)  -- одноразовые
e2ee_group_messages (id, chat_type, chat_id, kind SMALLINT,  -- welcome/commit/proposal
                     recipient_device UUID NULL, payload BYTEA, epoch BIGINT, created_at)
e2ee_membership_queue (id, group_id, action, target_user, created_at, processed_at NULL)
e2ee_backups        (user_id PK, blob BYTEA, kdf_params JSONB, updated_at)
```

### 6.3 Вложения
`attachments.data` — шифротекст (`crypto_secretstream`, шифруется до upload).
Имя/mime/размер-точный → в **зашифрованный метаблоб** внутри сообщения;
в открытых столбцах остаются только `id`, `size_ceil` (октеты шифротекста),
`created_at`. Ключ файла — случайный, передаётся внутри MLS-сообщения.

### 6.4 Что ломается и чем заменяется
| Серверная фича | Замена |
|---|---|
| `search_index` (tsvector по тексту) | Клиентский поиск по локальному кэшу (IndexedDB) |
| Текст в push-payload | Content-less push «новое сообщение», клиент подтягивает и расшифровывает |
| Серверный парсинг @упоминаний | Парсинг на клиенте, упоминания внутри шифротекста |
| Превью вложений на сервере | Локальная генерация превью до шифрования |

## 7. Границы E2EE (честная фиксация)

**Шифруется:** тексты сообщений 1:1 и групп, вложения, упоминания, имена
файлов, реплай-текст (сам `reply_to`-указатель открыт).

**НЕ шифруется (серверные бизнес-фичи, вне рамок E2EE):** доски (boards),
заметки (notes), голосования (polls), рейтинг, условия уровней — по ним сервер
исполняет логику (подсчёт голосов, права уровней) и не может работать вслепую.
Плюс метаданные: кто с кем, когда, размер, онлайн. Минимизация метаданных —
план §7, отдельный этап после основного. Всё это фиксируется в
`docs/security.md` при обновлении STRIDE.

## 8. Ответы на вопросы плана (§10)

1. **Групповой протокол — MLS** (ts-mls зрелая: тест-векторы, interop,
   активные релизы). Sender Keys не нужны. ✅ решено, §1.
2. **Прод-данные:** приложение ещё не деплоилось (deploy-ready, но не
   опубликовано) — **прод-данных нет, стартуем E2EE с чистой БД**, столбец
   `text` удаляется без ре-шифровки. Если это неверно — сказать до этапа 3.
3. **Серверный поиск по сообщениям — убираем**, замена клиентским поиском.
   Обязательный компромисс E2EE, принят этим документом.
4. **1:1-протокол — MLS-группа из двух**, вместо X3DH+DR (обоснование §1).
   FS и PCS сохраняются; регулярный Update-коммит (раз в N сообщений/дней)
   планируется в этапе 4.

## 9. Структура крипто-ядра (контракт для этапа 2)

`frontend/src/shared/crypto/` — не зависит от рантайма (web / Tauri webview /
Node-тесты), не импортирует ни React, ни `shared/api`:

```
shared/crypto/
  sodium.ts        — ленивая инициализация libsodium, типобезопасные обёртки
  keystore.ts      — IndexedDB-хранилище (non-extractable CryptoKey + шифрование
                     байтовых блобов локальным ключом); in-memory fallback для тестов
  identity.ts      — device identity (Ed25519), генерация, подпись/проверка ключей
  fingerprint.ts   — BLAKE2b-фингерпринты, safety number пары пользователей
  mls.ts           — обёртка над ts-mls: createChat / joinFromWelcome / encrypt /
                     decrypt / addMember / removeMember / update; сериализация
                     состояния групп в keystore
  attachments.ts   — secretstream-шифрование файлов (этап 6)
  backup.ts        — Argon2id + secretstream key bundle (этап 7)
  index.ts         — публичный интерфейс ядра
```

Интеграция (этапы 4+): шифрование в `entities/message` **до** `shared/api`/
`shared/ws`, расшифровка после получения до попадания в zustand-стор.

## 10. Критерии готовности E2EE (тесты-доказательства, план §5)

1. Строка `messages` в БД не содержит открытого текста; расшифровка без
   клиентского состояния невозможна.
2. Исключённый из группы не расшифровывает сообщения новых эпох.
3. Подмена key package сервером → клиент отвергает (подпись) или показывает
   баннер смены ключа (TOFU).
4. Push-payload и логи сервера не содержат открытого текста.
5. `grep`-аудит бэкенда: тело сообщения нигде не логируется.
