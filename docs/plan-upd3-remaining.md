## 0. Рамка безопасности (прочитать до первой строки кода)

За этапы E2EE 1–4 и UPD3 A–G у мессенджера сложилась защитная система, которую
**нельзя ломать**. Каждый оставшийся этап проектируется в этих границах:

1. **E2EE (MLS через ts-mls).** Личные чаты шифруются end-to-end: сервер хранит
   только `messages.ciphertext`, никогда открытый текст. Группы пока **не**
   E2EE (это E2EE-этап 5), вложения пока **не** шифруются (E2EE-этап 6). Сервер
   **не может** читать/парсить зашифрованный контент. **Forward secrecy:**
   MLS-ключи одноразовые, эпохи двигаются на add/remove участника; ciphertext
   привязан к эпохе.
2. **Клиентский кэш открытого текста.** Расшифрованный текст E2EE-сообщений
   лежит локально в IndexedDB-keystore под ключом `msg/<messageId>`
   (`frontend/src/entities/e2ee/chats.ts`). Всё, что «удаляется» на сервере,
   должно вычищаться и из этого кэша (критично для этапа J).
3. **Ролевая иерархия 1–10.** Меньший уровень = сильнее. У группы —
   `min_role_level` (широта аудитории). Нельзя раскрывать существование
   недоступного ресурса (маскированный 404); нельзя «поднимать» контент к более
   широкой аудитории (правило пересылки, этап D).
4. **Content-less push.** Пуш никогда не несёт тело сообщения. Гейтится mute и
   `notification_settings.group_mode` (этап G, `internal/notifprefs`).
5. **Без `dangerouslySetInnerHTML`.** Рендер через токенизацию (`shared/lib/richText`).
6. **SSRF-щит** на любом исходящем запросе сервера (`internal/linkpreview`).
   Новые этапы H–K исходящих запросов не добавляют — новой SSRF-поверхности нет.
7. **Аудит** привилегированных/необратимых операций (`internal/audit`).
8. **Аддитивность контрактов.** `SendMessageBody`/`Message` расширять только
   аддитивно; существующее поведение не ломать.

**Общий принцип для H–K:** сервер по-прежнему не читает шифротекст; всё, что
раньше «сервер смотрит в текст», делается либо на метаданных, либо на клиенте;
любое серверное удаление контента синхронно с клиентским кэшем.

Параллельный трек — **E2EE-роадмап (этапы 5–10 из `plan-e2ee-and-hardening.md`):
группы E2EE, шифрование вложений, QR-мультидевайс, клиентский поиск +
content-less push, Trusted Types/SRI, тесты-доказательства.** UPD3 H–K
проектируются так, чтобы быть forward-compatible с ними (см. пометки в этапах).

## Этап H — Папки и архив чатов

**Риск-профиль: низкий.** Папки/архив — это персональные организационные
метаданные поверх ссылок на чаты. Контент не трогается, E2EE не затрагивается,
исходящих запросов нет.

**БД (миграция `000033`).**
- `chat_folders (id, user_id, name, position, created_at)` — папки пользователя.
- `chat_folder_items (folder_id, chat_type, chat_id, PRIMARY KEY(folder_id, chat_type, chat_id))`.
- Архив: `chat_archives (user_id, chat_type, chat_id, archived_at, PRIMARY KEY(...))`
  — по образцу `chat_mutes` (этап G): архив у пользователя X виден только X.
  (Не поле в chats/groups — те таблицы общие, а архив персональный.)

**Бэкенд (`internal/chatfolders`, новый пакет по образцу `notifprefs`).**
- CRUD папок: `POST/PATCH/DELETE /folders`, переупорядочивание (`position`),
  `POST/DELETE /folders/{id}/items` (добавить/убрать чат).
- Архив: `PUT/DELETE /chats/{chatType}/{chatID}/archive` (по образцу mute).
- `GET /folders` — папки пользователя + их элементы; `GET /settings/archived`
  — список архивных (как `/settings/mutes`).
- **Клиренс/утечки:** элемент папки — просто ссылка на chat_id. Никакой проверки
  «а можно ли» при добавлении не нужно строго, НО отдача чатов всегда идёт через
  существующий доступ-фильтр `chatsSvc`/`groupsSvc`: недоступный чат в папке
  просто не появится в выдаче (папка не может «раскрыть» или «выдать» доступ).
  При добавлении в папку — мягкая проверка членства (можно вернуть 404-маску,
  если чат недоступен), чтобы не хранить мусор.
- `GET /chats` расширить аддитивно: без изменения дефолта. Фильтрация по папке и
  скрытие архивных делается на клиенте по данным `/folders` + `/settings/archived`
  (проще и не ломает существующий контракт), либо серверными query-параметрами
  `?folderId=&archived=` — выбрать одно, задокументировать.

**Фронтенд.**
- Вертикальная лента вкладок над/слева от списка чатов: «Все», «Непрочитанные»
  (клиентский фильтр по `unreadCount`), пользовательские папки.
- Контекст-меню/свайп чата: «В архив», «В папку …». Переиспользовать паттерн
  выпадающего меню из `MuteMenu` (этап G).
- Раздел «Архив» внизу списка; архивный чат при новом сообщении всплывает по
  настройке (можно завязать на существующий счётчик непрочитанного).
- Хуки `entities/chat-folders` (TanStack Query), инвалидция как в `notif-prefs`.

**Взаимодействие с прошлыми этапами.** Архив ортогонален mute (этап G): чат может
быть и замьючен, и заархивирован. Индикаторы не конфликтуют.

**Готово когда.** Папки создаются/переименовываются/сортируются; чаты
добавляются/убираются; архив/разархив работают; фильтры списка (Все/Непрочитанные/
папки) применяются; архив скрыт из основного списка, но доступен; недоступный чат
никогда не «протекает» через папку. OpenAPI + spec 09 обновлены; unit на CRUD/
переупорядочивание, integration на изоляцию по пользователю.

## Этап I — Отложенная отправка

**Риск-профиль: высокий (прямая коллизия с E2EE forward secrecy).** Смысл фичи —
сервер отправляет сообщение в `send_at` **без участия клиента** (устройство может
быть офлайн). Значит контент должен «дожить» до отправки на сервере.

**Ключевой тредофф E2EE (принять осознанно, зафиксировать в `docs/security.md`).**
Сервер не может зашифровать за клиента (у него нет ключей) и клиент не может
до-шифровать в момент отправки (он офлайн). Остаётся два пути:

- **Путь A — «шифруем заранее» (рекомендуется для 1:1).** Клиент в момент
  планирования шифрует MLS-сообщение текущей эпохи и кладёт `ciphertext` в
  `scheduled_messages`. Воркер в `send_at` прогоняет его через штатный
  send-пайплайн. **Риски и как их держать:**
  - *Дрейф эпохи.* Если к `send_at` эпоха группы сменилась (add/remove/self-update),
    получатели могли отпустить старые ключи (`keyRetentionConfig` ts-mls) →
    сообщение не расшифруется, у получателя покажется 🔒 (как «недоступная
    история», этап 4). Для личных чатов (2 участника, эпоха меняется редко —
    только периодический self-update) риск низкий. Честно предупредить в UI:
    «в зашифрованном чате отложенное сообщение может стать нечитаемым, если
    ключи чата обновятся».
  - *Ciphertext заранее на сервере.* Шифротекст лежит на сервере до `send_at` —
    это тот же класс данных, что и обычное E2EE-сообщение (сервер не читает),
    так что новых утечек нет; но сообщение «заморожено» (нельзя менять контент,
    только отменить/переслать заново).
- **Путь B — «только plaintext-чаты в v1».** Полная поддержка для групп (сейчас
  plaintext) и plaintext-fallback личных чатов; для E2EE-лички — либо запретить
  планирование (честный UI-дисейбл), либо путь A под предупреждением.

**Рекомендация:** реализовать путь A для личных чатов **с явным предупреждением**
и полный путь для групп; в `SendMessageBody`-совместимом снимке хранить либо
`text`, либо `ciphertext+alg+epoch`. Сервер шифротекст не читает.

**БД (миграция `000034`).**
`scheduled_messages (id, chat_type, chat_id, sender_id, text NULL, ciphertext BYTEA
NULL, alg SMALLINT NULL, content_kind SMALLINT NULL, reply_to UUID NULL,
attachment_ids UUID[] , send_at TIMESTAMPTZ, status VARCHAR CHECK(pending|sent|
canceled), created_at)`. Индекс на `(status, send_at)` под выборку воркера.

**Бэкенд (`internal/scheduled`, новый пакет).**
- `POST /messages/schedule`, `GET /messages/scheduled` (свои), `PATCH`/`DELETE`
  до отправки.
- **Доступ проверяется дважды:** при планировании (актор может писать в чат) и в
  `send_at` (членство могло измениться — если актор потерял доступ, запись
  переводится в `canceled`, не отправляется; это и защита от «поднятия»
  контента после потери доступа).
- **Воркер:** тикер + `SELECT … FOR UPDATE SKIP LOCKED`, прогон через штатный
  `messages.Send`, публикация `message.created`. **Идемпотентность:** статус
  `pending → sent` в одной транзакции; переживает рестарт, не отправляет дважды.
  Паттерн — как TTL-уборщик вложений (этап A) и будущий уборщик этапа J.
- **Вложения:** по id, ownership перепроверяется в `send_at` (файлы уже хранятся
  в БД). Если вложение удалено к моменту — отправить без него или отменить
  (решить, задокументировать).
- Push при отправке — content-less, через `notifications.OnMessage` (гейт mute/
  режима из этапа G применяется автоматически).

**Фронтенд.**
- В композере: долгий тап/стрелка у кнопки отправки → «Отправить позже» с выбором
  даты/времени. Для E2EE-лички — предупреждающий баннер (тредофф выше).
- Раздел «Запланированные» (в чате или настройках) — список с редактированием/
  удалением до отправки.
- Шифрование на клиенте перед планированием (путь A) переиспользует
  `entities/e2ee/encryptForChat`.

**Готово когда.** Сообщение уходит в назначенное время ровно один раз;
редактирование/отмена до отправки работают; переживает рестарт; для E2EE-лички
поведение (путь A + предупреждение) явно определено и протестировано; потеря
доступа к `send_at` → отмена без отправки. OpenAPI + spec; unit на воркер-логику
(идемпотентность), integration на планирование/отмену/доступ.

## Этап J — Исчезающие сообщения

**Риск-профиль: высокий, но усиливает security-позиционирование.** Здесь новая
система безопасности даёт **дополнительное требование, которого нет в исходном
мастер-промпте**: удаление должно затрагивать и клиентский кэш открытого текста.

**БД (миграция `000035`).**
- `messages.expires_at TIMESTAMPTZ NULL` (когда самоуничтожиться).
- `chat_disappear_settings (chat_type, chat_id, ttl_seconds NULL, set_by, updated_at)`
  — TTL по умолчанию для новых сообщений чата (общий для чата, не персональный:
  исчезающие сообщения — свойство чата, как в Signal/TG).
- Индекс `messages(expires_at) WHERE expires_at IS NOT NULL` под уборщик.

**Бэкенд.**
- `PUT /chats/{chatType}/{chatID}/disappearing {ttlSeconds|null}` — включает/
  выключает таймер по умолчанию. Доступ — членство в чате; **аудит** включения
  TTL (без содержимого).
- При отправке (если у чата задан ttl или у сообщения индивидуальный таймер) —
  `expires_at = now + ttl`. `expires_at` — метаданные (не контент), в E2EE-чате
  серверу видны и это нормально.
- **Воркер-уборщик** (паттерн этапа I/A): **жёстко** удаляет истёкшие строки
  `messages` и связанные `attachments` (и `ciphertext`, и байты файлов), шлёт
  `message.deleted`. Это **системное** удаление — не нарушает правило «пользователь
  не может хардтелить чужое»; обычные пользователи по-прежнему не удаляют чужое
  вручную.
- Взаимодействие с этапом D (пересылка): форвард — новое сообщение; наследует
  **TTL целевого** чата (не источника). С этапом I: отложенное в исчезающий чат
  получает `expires_at = send_at + ttl`.

**Фронтенд — критично для безопасности.**
- Переключатель таймера в шапке/настройках чата (выкл / 1ч / 24ч / 7д / кастом) +
  опция на отдельном сообщении в контекст-меню.
- Индикатор таймера и обратный отсчёт на исчезающих пузырях; аккуратное удаление
  из ленты по `message.deleted`.
- **Обязательно:** при `message.deleted` для исчезающего (и вообще любого)
  сообщения — вычищать локальный кэш открытого текста E2EE
  (`s.store.remove('msg/<id>')`) и запись в возможном клиентском поисковом кэше.
  Иначе «исчезнувший» текст останется в IndexedDB и фича безопасности будет
  дырявой. Это новое требование из-за E2EE-кэша (п.2 рамки).
- Совместимость с этапом 8 E2EE-роадмапа (клиентский поиск по расшифрованному
  кэшу): поисковый индекс должен уважать удаление — заложить хук чистки.

**Готово когда.** TTL на чат и на сообщение работает; истёкшее реально удалено на
сервере (текст, ciphertext, файлы) и у клиентов, **включая клиентский plaintext-
кэш**; таймер/анимация на месте; аудит включения TTL пишется; E2EE не нарушен.
Тесты: integration на уборщик (истёкшее удаляется, вложения тоже), unit/e2e на
клиентскую чистку кэша.

## Этап K — Треды/обсуждения в группах (последним — самый инвазивный)

**Риск-профиль: средний.** Только группы (сейчас plaintext, не E2EE). Спроектировать
**forward-compatible** с будущими E2EE-группами (E2EE-этап 5): тред — это
метаданные (`thread_root_id`) поверх того же потока сообщений; когда группа станет
MLS-группой, тред-сообщения будут обычными application-сообщениями той же
MLS-группы, без отдельной крипты.

**БД (миграция `000036`).**
- `messages.thread_root_id UUID NULL` (корень ветки).
- Денормализация на корневом сообщении: `thread_reply_count INT DEFAULT 0`,
  `thread_last_reply_at TIMESTAMPTZ NULL`.
- Индексы: под выборку ветки `(thread_root_id, created_at)` и под сортировку
  корней по последнему ответу.

**Бэкенд (`internal/messages`).**
- Отправка в тред: `SendMessageBody`/`POST /messages` расширить аддитивно полем
  `threadRootId` (валидировать: корень существует и в том же чате).
- `GET /messages/thread?rootId=&cursor=` — сообщения ветки, **та же проверка
  доступа к группе**, что и основной список (никаких перекрёстных утечек;
  клиренс наследуется).
- Счётчики ветки обновлять **транзакционно** при отправке ответа; `message.created`
  нести `threadRootId`, чтобы клиенты обновили и ленту, и открытую панель.
- Основная лента группы: корни + «N ответов», без дублирования ответов ветки в
  основном потоке (фильтр `thread_root_id IS NULL` в основном `ListPage`).
- **Поиск** по треду скоупится так же, как основной (существующий клиренс-скоуп
  `internal/search` / `internal/chatmedia`).
- **E2EE-совместимость:** `thread_root_id` — метаданные, серверу видны и в
  будущей E2EE-группе (как `reply_to` сейчас). Тело ответа шифруется/не шифруется
  ровно как обычное сообщение группы. Ничего специфичного для крипты добавлять
  не нужно.

**Фронтенд (`widgets/conversation`, прежде всего `GroupView`).**
- Плашка «N ответов · время последнего» под корневым сообщением; клик открывает
  панель треда (правая колонка/overlay) с собственным композером (переиспользовать
  `Composer` с проставленным `threadRootId`).
- Реалтайм-обновление счётчиков и открытой панели по WS.
- Совместимость с этапом F (reply-jump): reply внутри треда остаётся в треде.
- Совместимость с этапом G: пуш по тред-ответам уважает mute/`group_mode`.

**Готово когда.** Ответы в тред не засоряют основную ленту; счётчики и «последний
ответ» обновляются в реалтайме; панель треда открывается/пишет/скроллит; доступ
ограничен участниками группы; поиск скоупится корректно; всё покрыто тестами.

## Порядок и сквозные требования

1. **Строго по очереди H → I → J → K** (риск/инвазивность растут). После каждого —
   сборка, `go vet`/`tsc`/линтеры, `go test`/фронт-тесты/интеграция с реальным
   Postgres (docker), `docker compose up --build` с нуля, стоп на ревью.
2. **Миграции 000033–000036**, обратимый `down` у каждой.
3. **E2EE не ломать:** сервер не читает шифротекст ни на одном этапе; удаление
   контента (этап J) синхронно с клиентским кэшем `msg/<id>`.
4. **Ролевая иерархия:** никаких утечек существования/доступа (папки, треды,
   отложенные, исчезающие).
5. **Push (этап G) и аудит** применяются к новым путям доставки (отложенные,
   тред-ответы); аудит — на включение TTL (J) и, при желании, на операции с
   доступом.
6. **Контракты аддитивны:** `threadRootId` (K), snapshot полей (I) — без слома
   `SendMessageBody`/`Message`.
7. После K — обновить `docs/security.md` (STRIDE) и прогнать `/security-review`.

## Связь с E2EE-роадмапом (остаток 5–10)

Не входит в UPD3, но взаимодействует:
- **E2EE-этап 5 (группы MLS):** делает этап K тред-сообщения зашифрованными
  «бесплатно» (метаданные `thread_root_id` остаются в открытом виде).
- **E2EE-этап 6 (вложения):** влияет на этапы I/J (шифрованные вложения тоже
  должны доживать до `send_at` и жёстко удаляться по `expires_at`).
- **E2EE-этап 8 (клиентский поиск):** этап J обязан чистить клиентский поисковый
  индекс при исчезновении — заложить хук уже сейчас.
- **E2EE-этап 9 (Trusted Types/SRI):** не конфликтует с H–K.

Приоритезацию (сначала докрутить E2EE 5–10 или сначала UPD3 H–K) — **уточнить у
заказчика**; по умолчанию продолжаем UPD3 H–K, помечая точки будущей интеграции.
