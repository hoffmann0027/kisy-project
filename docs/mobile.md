# Mobile / адаптив — правила и чек-лист

Кратко о том, как KISY ведёт себя на телефоне, чтобы новые компоненты не
ломали адаптив. Правки внесены этапами M1–M6; ниже — инварианты, которые
нужно держать.

## Точка перелома

- **760px** — единственный основной брейкпоинт (`@media (max-width: 760px)`):
  сетка мессенджера схлопывается в одну колонку, rail становится нижней
  панелью, модалки превращаются в bottom sheet. При нужде — вторичный 480px.
- Аналитика/доски рейтинга дополнительно ужимаются в один столбец при 900px.

## Высоты: только `dvh`, не `vh`

На мобильных `100vh` — это «большой» viewport (как если бы адресная строка
была скрыта), поэтому низ блока и его внутренний скролл недостижимы при
видимой хромке. Правило:

- Для высот/`max-height`, влияющих на видимую область, использовать **`dvh`**.
- Фолбэк для старых браузеров — как в `shared/config/theme.css`: базовое
  правило в `vh`, затем переопределение в `@supports (height: 100dvh) { … }`.
- `html, body, #root` уже на `100dvh` под `@supports`.

## Safe-area (вырез, home indicator)

- `frontend/index.html` содержит `viewport-fit=cover` — **без него**
  `env(safe-area-inset-*)` возвращает 0.
- Токены в `theme.css`: `--safe-top/right/bottom/left = env(safe-area-inset-*,
  0px)`. Применять только через `calc()`/`max()`, тогда на устройствах без
  выреза (и на десктопе) инсеты = 0 и вычисления схлопываются к прежним
  значениям — десктоп не меняется.
- Фиксированные/абсолютные оверлеи (backdrop модалки, экраны звонков, тосты,
  нижний rail) обязаны учитывать соответствующие инсеты.

## Экранная клавиатура

- Хук `shared/lib/useVisualViewport.ts` (подписка на `window.visualViewport`)
  публикует высоту клавиатуры в токен **`--kb-inset`** на `<html>` (0 на
  десктопе и до гидрации).
- Поднимать композер/поля через **inset/height** (вычитать `--kb-inset` из
  высоты контейнера, поднимать `bottom`), **не** через `position: fixed` — на
  iOS это даёт артефакты.
- Модалки: `max-height` с учётом `--kb-inset`, активное поле само въезжает в
  видимую зону (см. `Modal.tsx`).

## Rail (нижняя навигация)

- Геометрия — глобальные токены в `theme.css` (rail рендерится и в мессенджере,
  и на странице рейтинга): `--rail-h` (полоса кнопок), `--rail-pad-bottom`
  (растёт на home indicator, гасится клавиатурой), `--rail-total`.
- Высоты колонок контента вычитают `--rail-total` (и `--kb-inset`, где нужно).

## Тач

- **Тач-цели ≥ 44×44px** на мобиле (аудит по `width: 30/38px` и подобным).
- `touch-action` — **точечно**. Пример: на карточке доски `touch-action:
  pan-y` (вертикальный свайп = скролл колонки, горизонтальный = перенос).
  **Никогда** не вешать `touch-action: none` на контейнер со скроллом.
- Перетаскивание — на **Pointer Events** (`onPointerDown/Move/Up`), не на
  нативном HTML5 DnD: последний на мобильных браузерах не срабатывает вовсе
  (см. `widgets/board/useCardDrag.ts`).

## Никакого горизонтального выезда

- Ни один экран не должен скроллиться вбок (`scrollWidth === clientWidth`).
- Широкие таблицы/сетки — горизонтальный скролл **внутри своего контейнера**
  (`overflow-x: auto` + `min-width` на таблице), не страницы (см. `admin.css`).

## Чек-лист ручной проверки на устройстве (DoD этапов)

Прогнать на **iPhone (Safari + установленный PWA)** и **Android (Chrome)**,
ширины **360 / 390 / 414**, портрет и ландшафт. По каждому экрану:

- [ ] Скролл: списки, переписка, тело модалки прокручиваются до конца.
- [ ] Клавиатура: композер и активное поле видны над клавиатурой.
- [ ] Safe-area: нижний rail и тосты выше home indicator; верх не под вырезом;
      в установленном PWA шапка не лезет под статус-бар.
- [ ] Тач-цели ≥44px; нет промахов.
- [ ] Нет горизонтального выезда страницы.
- [ ] Доска: карточка перемещается пальцем; вертикальный скролл колонки и
      горизонтальное листание не конфликтуют с переносом; тап открывает карточку.
- [ ] Календарь: свайп смены месяца не конфликтует с вертикальным скроллом.
- [ ] Темы (glass/luce/aurora/cyber/xp/matrix) не сломаны.

## Автотесты

- Юнит: `useVisualViewport.test.ts` покрывает математику инсета клавиатуры,
  порог отсечки хромки и отписку.
- E2E-фреймворка в проекте нет; при появлении Playwright/Cypress — добавить
  прогон в мобильном viewport (скролл переписки + открытие модалки-sheet).
