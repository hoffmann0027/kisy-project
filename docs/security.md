# KISY — Модель безопасности

Этот документ фиксирует модель угроз (STRIDE) и контроли, реализующие
`docs/spec/06-security.md`. Это опорный документ для прохода по
security-hardening и для будущих аудитов.

## Принципы

Глубоко эшелонированная защита (defense in depth), наименьшие привилегии,
безопасное по умолчанию, zero-trust между слоями. Каждое привилегированное
действие аудируется; внутренние ошибки никогда не показываются клиентам.

## Модель угроз STRIDE

### Spoofing (подмена личности)
- **Угрозы:** кража учётных данных, перехват сессии, поддельная личность.
- **Меры:** хеширование паролей Argon2id (`internal/auth/password`);
  access-токены JWT, подписанные HS256 секретом ≥32 символов; opaque
  refresh-токены, хранящиеся только как SHA-256 дайджесты; HTTPOnly +
  `SameSite=Strict` cookie; валидация сессии на каждый запрос (logout и
  смена пароля вступают в силу немедленно); блокировка аккаунта после 5
  неудач / 15 мин; ограничение частоты по IP (Redis + Nginx). Логин с
  выровненным временем не позволяет перечислять пользователей.

### Tampering (несанкционированное изменение)
- **Угрозы:** подмена запроса/ответа, SQL-инъекции, подделка сообщений.
- **Меры:** параметризованные запросы везде (pgx, без конкатенации строк);
  строгое декодирование JSON с отклонением неизвестных полей и лимитом тела
  1 МиБ (`pkg/httpjson`); ротация refresh-токенов с детекцией повтора
  (повторно предъявленный токен отзывает сессию); TLS 1.3 при передаче;
  append-only `audit_logs` (UPDATE/DELETE заблокированы триггером БД).

### Repudiation (отказ от совершённого действия)
- **Угрозы:** пользователь или админ отрицает совершённое действие.
- **Меры:** неизменяемый журнал аудита, фиксирующий актора, действие, цель,
  хешированный IP, идентификатор сессии, идентификатор запроса и метку
  времени для каждой привилегированной операции (вход, выход, регистрация,
  создание/использование приглашения, смена роли, сброс пароля,
  активация/деактивация, удаление сообщения, создание группы, security-
  события повтора refresh-токена). Читается только CEO через `GET
  /admin/audit`.

### Information Disclosure (раскрытие информации)
- **Угрозы:** утечка данных или факта существования ресурсов между
  уровнями допуска.
- **Меры:** модель допуска (`internal/access`) — группы выше допуска
  пользователя невидимы и возвращают **404, а не 403**, поэтому их
  существование нельзя проверить; ошибки доступа к сообщениям/чатам
  сводятся к not-found; директория пользователей возвращает только аккаунты
  равного или более низкого допуска; пароли/refresh-токены никогда не
  покидают бэкенд; IP-адреса хранятся в виде посоленного хеша; внутренние
  ошибки возвращают обобщённое сообщение с идентификатором запроса, никогда
  не стек-трейс.
- **UPD3 (этапы H–K), тот же принцип на новых поверхностях:** папки/архив —
  персональные метаданные, добавление недоступного чата даёт маскированный
  404, а лента всегда строится из access-фильтрованных списков (папка не
  может «раскрыть» чат); отложенная отправка — доступ проверяется при
  планировании И повторно в send_at с актуальной ролью (потеря доступа →
  отмена с затиранием контента, никакого «поднятия» контента); треды —
  доступ к ветке равен доступу к группе, чужой корень неотличим от
  несуществующего; таймеры исчезновения и thread_root_id — метаданные:
  сервер по-прежнему не видит контента E2EE-сообщений.

### Denial of Service
- **Угрозы:** брутфорс, флуд запросами, slowloris, чрезмерные полезные
  нагрузки.
- **Меры:** зоны `limit_req` в Nginx (auth 5 r/s, api 30 r/s на IP) плюс
  per-IP лимиты в бэкенде на Redis; блокировка аккаунта; таймауты
  заголовков/тела и `client_max_body_size`; таймаут чтения заголовков
  сервера; лимиты чтения WebSocket и отбрасывание кадров медленным
  потребителям; корректное завершение (graceful shutdown).

### Elevation of Privilege (повышение привилегий)
- **Угрозы:** пользователь с низким допуском получает более высокий доступ.
- **Меры:** RBAC-мидлварь (`RequireAuth` + `RequireClearance`), 1 = CEO;
  админ-поверхность закрыта на уровне допуска 1; CEO не может понизить или
  деактивировать сам себя (нет самоблокировки); правило инициации личного
  чата запрещает обращаться «наверх»; уровень роли встроен в валидируемый
  access-токен и повторно сверяется с живой сессией на каждый запрос.

## Контроли application-security

| Атака                | Контроль                                                             |
|----------------------|---------------------------------------------------------------------|
| XSS                  | Строгий CSP (`script-src 'self'`, без inline-скриптов); экранирование React |
| CSRF                 | Cookie `SameSite=Strict` + мидлварь проверки Origin/Referer         |
| Clickjacking         | `X-Frame-Options: DENY` + CSP `frame-ancestors 'none'`              |
| SQL-инъекции         | Только параметризованные запросы pgx                                 |
| MIME-sniffing        | `X-Content-Type-Options: nosniff`                                    |
| Открытые редиректы   | Нет управляемых пользователем редиректов; `form-action 'self'`, `base-uri 'self'` |
| Path traversal / IDOR| UUID-ключи; проверки владения/допуска на ресурс; маскировка not-found |
| Утечка в ошибках     | Обобщённые ошибки клиенту, структурированные логи на сервере        |
| Понижение транспорта | HSTS (2 года, preload) + только TLS 1.3 + редирект HTTP→HTTPS 308   |

## Транспорт и заголовки

- **TLS:** в проде терминация только TLS 1.3 на edge-Nginx
  (`deploy/nginx/nginx.tls.conf`), cipher-suite с встроенным PFS, HSTS с
  preload, `http2`. Dev-сертификаты через `scripts/gen-dev-certs.sh`;
  включается `docker compose -f docker-compose.yml -f docker-compose.prod.yml up`.
- **Заголовки:** CSP, `X-Content-Type-Options`, `X-Frame-Options`,
  `Referrer-Policy`, `Permissions-Policy`, `Cross-Origin-*` ставятся
  бэкендом (`internal/platform/security`) на API-ответы и Nginx на
  статику; `server_tokens off` скрывает версию.

## Секреты

Все секреты (пароли БД/Redis, JWT-ключи, соль для IP-хеша, bootstrap-креды
CEO) берутся из переменных окружения / `.env`, никогда не хардкодятся.
`.env` и приватные TLS-ключи в `.gitignore`.

## Голосовые звонки (WebRTC)

- **Авторизация сигналинга:** сервер ретранслирует `call.*` только между
  участниками общего direct-чата (переиспользуется проверка участника чата);
  отправитель любого кадра должен входить в этот `callId`. Неавторизованный/
  битый сигналинг → обобщённый `error`, без утечки деталей.
- **Rate limiting:** инициация звонков ограничена (10/мин на пользователя),
  REST `/calls/*` — 60/мин на IP.
- **TURN-креды эфемерные:** выдаются на время (HMAC-SHA1 от `TURN_SECRET`),
  сам секрет остаётся на сервере; coturn запрещает relay в приватные/loopback
  диапазоны (`denied-peer-ip`).
- **Микрофон:** `Permissions-Policy: microphone=(self)` — доступ только
  своему источнику; камера/гео/платежи по-прежнему запрещены. `getUserMedia`
  требует secure-context (в проде — HTTPS; локально — `localhost`).
- **Аудит:** старт/ответ/отклонение/отмена/завершение/таймаут пишутся в
  аудит-лог; каждый звонок фиксируется в `call_logs`. Медиа идёт P2P и через
  сервер не проходит.

## Отложенная отправка и E2EE (UPD3 этап I — «путь A»)

Смысл фичи — сервер отправляет сообщение в `send_at` без участия клиента
(устройство может быть офлайн). Для E2EE-лички это создаёт принципиальный
тредофф: сервер не может зашифровать за клиента (у него нет ключей), а
клиент не может до-шифровать в момент отправки (он офлайн). Принятое
решение — **путь A, «шифруем заранее»**:

- Клиент шифрует MLS-сообщение **в момент планирования** и кладёт
  `ciphertext` (+ `alg`, `epoch`) в `scheduled_messages`. Сервер хранит и
  переносит только шифротекст — тот же класс данных, что у обычного
  E2EE-сообщения; новых утечек нет. Открытый текст кэшируется локально под
  `sched/<id>` и перевешивается на `msg/<id>` при доставке (MLS-отправитель
  не может расшифровать собственный шифротекст — ключи одноразовые).
- **Дрейф эпохи:** если к `send_at` эпоха чата сменилась (add/remove/
  self-update участника), получатель мог отпустить старые ключи и сообщение
  придёт нечитаемым (🔒, как «недоступная история»). Для лички (2 участника)
  риск низкий; UI честно предупреждает об этом при планировании.
- **Заморозка снапшота:** запланированное E2EE-сообщение нельзя изменить на
  сервере — только заменить новым шифротекстом с клиента или отменить.
  Отмена (пользователем или системой) удаляет шифротекст с сервера.
- **Доступ проверяется дважды:** при планировании и повторно в `send_at` с
  актуальным уровнем роли; потеря доступа/деактивация отправителя →
  `canceled`, содержимое затирается, ничего не отправляется. Это же
  запрещает «поднятие» контента в чат, к которому отправитель потерял
  доступ.
- **Ровно один раз:** воркер вставляет сообщение и переводит строку
  `pending → sent` в одной транзакции (`FOR UPDATE SKIP LOCKED`); рестарт в
  любой точке не даёт ни дубля, ни потери. Push при отправке — content-less
  и проходит штатный гейт mute/`group_mode` (этап G).

## Исчезающие сообщения (UPD3 этап J)

Таймер (`expires_at`, TTL чата в `chat_disappear_settings`) — метаданные:
в E2EE-чате сервер видит, когда сообщение умрёт, но не что в нём было.
Гарантии удаления:

- **Сервер:** уборщик жёстко удаляет истёкшие строки `messages` — текст,
  шифротекст и байты вложений уходят каскадом, полнотекстовый индекс
  чистится явно. Это системное удаление: право пользователей на удаление
  чужих сообщений не расширяется.
- **Клиент (критично из-за E2EE-кэша):** на каждый `message.deleted` —
  и исчезновение, и обычное удаление — клиент вычищает локальный кэш
  открытого текста (`msg/<id>` в IndexedDB-keystore). Без этого
  «исчезнувший» E2EE-текст пережил бы удаление на устройстве и фича была
  бы дырявой. MLS-ключи одноразовые, поэтому после чистки шифротекст
  невосстановим на устройстве. Сюда же встанет чистка клиентского
  поискового индекса (E2EE-этап 8) — хук уже в `dropPlaintext`.
- **Офлайн-гарантия (не полагаемся только на WS-событие):** WS доставляет
  `message.deleted` лишь подключённым клиентам, поэтому кэш открытого
  текста хранит `expires_at` рядом с самим текстом. Запись само-вытесняется
  при чтении (`cachedPlaintext` не отдаёт истёкшее и удаляет его) и
  периодическим свипом (`sweepExpiredPlaintext` при инициализации сессии и
  раз в час). Даже если устройство было офлайн в момент серверной уборки,
  расшифрованный текст не переживёт свой таймер локально.
- Событие исчезновения несёт `expired: true` — пузырь убирается из ленты
  целиком (без надгробия «сообщение удалено»).
- Включение/смена/выключение TTL чата аудируется (только сам TTL).
- Пересылка наследует TTL **целевого** чата; отложенное сообщение в
  исчезающий чат получает `expires_at = send_at + ttl` (таймер ставится в
  момент фактической отправки).

## Гигиена зависимостей и цепочка поставок

`govulncheck ./...` (Go) и `npm audit` (frontend, shipped-зависимости)
проходят чисто (0 уязвимостей) и включены в CI. Дополнительно (июль 2026):

- **`frontend/.npmrc`:** `ignore-scripts=true` (блокирует postinstall/
  preinstall — основной вектор компрометаций 2025–2026: axios, node-ipc,
  Shai-Hulud), `min-release-age=7` (карантин свежих версий), `save-exact`.
- **Dependabot:** `cooldown: 7 дней` на все экосистемы; PR ревьюятся вручную,
  auto-merge намеренно НЕ включается.
- **CI security-джоб:** `gitleaks` (секреты по всей истории), `trivy config`
  (Dockerfile/IaC), `gosec` (SAST Go), SBOM (`trivy fs --format cyclonedx`)
  как артефакт. Джоб `docker` дополнительно гоняет `trivy image` (CRITICAL,
  fixed-only). Образы сканеров запинены по digest.

## Обновление безопасности (июль 2026)

Проход по итогам аудита `KISY_security_update_2026_1`:

- **Cross-Site WebSocket Hijacking (fail-open Origin):** `ws/handler.go`
  теперь fail-closed — хендшейк принимается только при совпадении Origin с
  настроенным `WS_ALLOWED_ORIGIN` **или** с Host запроса (общая функция
  `security.OriginAllowed`, та же, что и в CSRF). Пустой `WS_ALLOWED_ORIGIN`
  больше не пропускает произвольные origin.
- **Fail-closed production guard (`config.validateProduction`):** при
  `APP_ENV=production` процесс отказывается стартовать, если пуст
  `WS_ALLOWED_ORIGIN`, `POSTGRES_SSLMODE=disable` по не-loopback соединению
  (без явного `ALLOW_PLAINTEXT_DB=true` для одно-хостовой приватной сети),
  пуст `REDIS_PASSWORD`, либо любой из секретов совпадает с известно-
  скомпрометированным значением (дайджесты в `knownCompromisedDigests`).
- **Ротация секретов:** значения из переданного за пределы среды `.env`
  считаются сожжёнными и ротированы; guard навсегда отклоняет их в проде.
- **Обязательная смена seed-пароля CEO:** bootstrap создаёт CEO с
  `must_change_password=true`; флаг отдаётся в `/me` и login, фронт
  (`RequireAuth` → `ForcePasswordChange`) блокирует приложение до смены.
- **Container hardening (`docker-compose.prod.yml`):** `read_only` + tmpfs +
  `cap_drop: [ALL]` (+ точечные `cap_add`) + `no-new-privileges` + лимиты
  ресурсов; фронтенд переведён на `nginx-unprivileged` (non-root, порт 8080).

## Отложено на будущие этапы (roadmap)

- **Завершить переход сообщений на MLS-ciphertext и дропнуть колонку `text`.**
  Приоритет №1 по конфиденциальности: пока переключение не завершено, БД
  содержит открытый текст, и компрометация БД = утечка переписки.
- **Шифрование вложений** тем же MLS-контекстом (сейчас `attachments.data`
  BYTEA в открытом виде) и переезд BYTEA → объектное хранилище (MinIO, SSE,
  presigned URL с коротким TTL, ClamAV на загрузках).
- **Шифрование at rest** для БД и бэкапов; автоматические шифрованные
  offsite-бэкапы (`age`/`gpg`) с регулярно тестируемым восстановлением.
- **Пост-квантовый MLS:** гибридный cipher suite (X25519MLKEM768) — когда
  клиентская MLS-библиотека поддержит; защита от harvest-now-decrypt-later.
- **Изоляция секретов:** Docker secrets/Vault вместо `.env` в проде.
- Сканирование загружаемых файлов на вредонос (таблица `attachments` уже
  гейтит видимость по `scan_status = clean`).
- CAPTCHA после многократных неудач (блокировка + rate limiting покрывают
  базовый уровень).
